# 08. Архитектурные Заметки, Ограничения И Риски

## Назначение Главы

Эта глава завершает книгу не перечислением файлов, а инженерными выводами.
Её задача — зафиксировать:
- что в архитектуре уже выглядит сильным;
- что пока остаётся нестабильным;
- где есть риск путаницы или технического долга;
- куда логично развивать проект дальше.

Такой раздел важен, потому что чистое описание текущего состояния не всегда показывает, где архитектура уже хороша, а где она только намечена.

## Сильные Стороны Архитектуры

### 1. Чёткое разделение слоёв

Это, пожалуй, самая заметная сильная сторона проекта.
У него уже есть отдельные зоны:
- сборка;
- контракты;
- исполняемый код;
- документация;
- рабочая лаборатория.

Такое разделение резко повышает управляемость системы.

### 2. Сильный контрактный слой `Includes`

Во многих проектах include-файлы являются хаотичным побочным продуктом.
Здесь они играют роль полноценного словаря системы.
Это делает код более анализируемым и облегчает масштабирование архитектуры.

### 3. Модульность рантайма

`Core`, `MainMenu`, `Session`, `World` выражены как крупные runtime-состояния с явной моделью загрузки и запуска.
Это создаёт чистую основу для развития проекта.

### 4. Грамотное разделение данных мира и AI-памяти

`FCharacter`, `FObjectCharacter`, `FAIContext`, `FBlackboard`, `FStateTreeContext` уже разложены по ответственности достаточно аккуратно.
Это предотвращает хаотическое смешивание source-of-truth данных и временных AI-решений.

## Ограничения, Которые Уже Видны

### 1. Документация проекта только начинает догонять реальную сложность системы

Сам репозиторий уже большой и структурный, а документация до последнего времени была обзорной и фрагментарной.
Создание этой книги — шаг к исправлению ситуации, но её ещё предстоит углублять дальше.

### 2. Не все runtime-потоки одинаково детально выражены в документации

`World` и `StateTree` уже читаются лучше, чем некоторые другие участки.
Например, по мере роста проекта имеет смысл отдельно документировать:
- rendering pipeline;
- object update pipeline;
- asset manager;
- сохранения и загрузку сессии.

### 3. `Blackboard` пока не стабилизирован как модель

Слот выделен, архитектурное место определено, но окончательный semantic contract ещё не закрыт.
Это нормально, но важно не оставить такую структуру навсегда как “dummy payload без правил”.

### 4. ID-ссылки требуют строгой дисциплины

Модель через `CharacterID`, `ObjectID`, `ParticipantID`, `AIContextID` компактна и правильна для платформы,
но любое перемещение структур по массивам требует внимательного обновления ссылок.
Если за этим не следить системно, целостность данных будет первой зоной поломки.

## Места, Где Особенно Важна Дисциплина Именования

### `Character`, `ObjectCharacter`, `ObjectCharacterAI`

Сейчас модель уже движется в правильную сторону, но для большого проекта особенно важно, чтобы имена структур однозначно показывали слой ответственности:
- данные персонажа;
- world-object;
- AI-runtime.

## Главные Архитектурные Правила, Которые Стоит Сохранить

### Правило 1. Не смешивать source of truth и рабочую память

Если данные являются канонической частью мира или персонажа, они должны жить в своей сущности.
AI не должен становиться вторым хранилищем тех же фактов.

### Правило 2. Не зашивать AI в базовый объект без необходимости

`FObjectCharacterAI` как специализированное расширение выглядит правильнее, чем принудительное хранение AI-полей во всех world-объектах.

### Правило 3. Держать `Descriptor` и runtime отдельно

Это правило уже соблюдается в `StateTree`, и его нужно сохранить любой ценой.
Именно оно даёт модели ясность и масштабируемость.

### Правило 4. Сохранять `Includes` как единый словарь системы

Не стоит растворять ключевые структуры, константы и layout-память в `Source/`.
Сила текущей архитектуры именно в том, что контрактный слой существует отдельно.

### Правило 5. Развивать документацию вместе с кодом

Чем глубже становится система, тем дороже потом восстанавливать её замысел постфактум.
Поэтому книга должна обновляться параллельно с развитием архитектуры.

## Что Логично Документировать Дальше

Эта книга уже задаёт скелет, но для очень глубокого уровня анализа следующими логичными темами будут:
- файл-за-файлом разбор `Source/Modules/World/`;
- отдельная книга по rendering pipeline;
- отдельная книга по asset manager и memory allocation;
- отдельная книга по save/load pipeline;
- отдельная книга по object update и sorting;
- таблица всех глобальных runtime-переменных;
- карта page memory и deploy-блоков в виде отдельных схем.

## Финальный Вывод По Архитектуре Проекта

Проект уже обладает всеми признаками серьёзной системной структуры:
- отдельный build-layer;
- отдельный contract-layer;
- модульный runtime;
- предметная модель данных;
- формирующийся AI-layer;
- выраженное мышление через layout памяти и page-based исполнение.

Самая сильная сторона текущего состояния проекта — он уже структурирован лучше, чем документирован.
Это хорошая проблема.
Она означает, что у архитектуры есть реальная форма, и её можно последовательно описывать, а не выдумывать задним числом.

Созданная книга — это не конечный результат, а правильный контейнер для дальнейшего углубления.
Теперь проект можно документировать не как россыпь заметок, а как систему глав, в которых каждая отвечает за свой слой ответственности.
